# Архитектура: админка, БД, бот

Краткая модель Via для эксплуатации и разработки. Детальная матрица экран → таблица → бот: [ADMIN_DB_BOT_AUDIT.md](ADMIN_DB_BOT_AUDIT.md).

## Принцип

- **Эксплуатация:** оператор не обязан править код репозитория. Политика «кому, куда, с какими фильтрами», справочники, шаблоны, интервалы из UI, секреты в Postgres; бот подхватывает данные при старте и при hot reload (см. [ADMINISTRATOR_GUIDE.md](ADMINISTRATOR_GUIDE.md)).
- **Продукт:** смена семантики уведомлений и пайплайна — в коде `src/bot/` и релиз.
- **Инфраструктура:** `.env` / compose (`ADMIN_URL`, `BOT_INSTANCE_ID`, логи, часть таймингов без UI) — зона деплоя.

Цель «всё настраивается в админке» для повседневной работы достигается за счёт БД и UI; полный вынос алгоритма в конфиг (rule engine) — отдельный крупный эпик, не описан здесь.

## Что не в «настройках в таблицах»

Сознательно не дублируется в SQL:

- Журнал **«События»** — чтение файла лога на диске (см. гайд по аудиту).
- **Статус контейнера бота** на дашборде — Docker и при необходимости `runtime_status.json`.

## Оставшиеся параметры только в env

Часть интервалов и лимитов по умолчанию задаётся в [`src/config.py`](../src/config.py) и при необходимости вынесена в `cycle_settings` по мере появления полей в UI. Если параметр есть только в `.env`, это отражено в комментариях `config.py` и в гайде администратора (раздел про рестарт и hot reload).
