# План: онбординг подписчиков через личку Matrix-бота

Цель: сотрудники **не заходят в веб-панель**; подписка на уведомления настраивается **диалогом с ботом** в личной комнате (DM).

**Разделение контуров:** админка — инфраструктура, маршруты, секреты сервиса; Matrix-бот — самообслуживание подписчиков (минимум привилегий в панели).

## Текущее состояние

- Панель — в основном для **админов**: пользователи, группы, маршруты, секреты.
- Страница **«Привязка Matrix»** (код в комнате + подтверждение в UI) **удалена** из продукта.
- Таблица `matrix_room_bindings` в БД может оставаться от старых миграций (не используется админкой).

## Целевой сценарий (MVP)

1. Организация сообщает сотрудникам **MXID бота** (например `@support_token_bot:server`).
2. Сотрудник находит бота в поиске Matrix и пишет любое первое сообщение → создаётся **личная комната** с ботом.
3. Бот отвечает приветствием и **инструкцией**, какие данные прислать (пошагово или одним блоком — на выбор реализации):
   - API-ключ Redmine (см. «Безопасность» и «Логи, удаление сообщений, хеш vs шифрование»);
   - числовой **ID пользователя** в Redmine (проверка согласованности с ключом);
   - **отдел** (строка → `support_groups` по имени);
   - **дни недели** и **часы** (таймзона из `BOT_TIMEZONE` / МСК по политике);
   - **типы уведомлений** («все» или перечень как в `bot_users.notify`).
4. Команды **`!change`**, **`!cancel`**, при необходимости **`!help`** / **`!status`**.

**Matrix ID пользователя:** обычно **не спрашивать в чате** — брать **`sender`** из события (`@user:server`). Спрашивать имеет смысл только в нестандартных случаях (несколько аккаунтов, делегирование); тогда это отдельный шаг с валидацией.

## Логи, удаление сообщений, хеш vs шифрование

### Можно ли не логировать передачу API-ключа и Redmine ID?

**Да, для кода это обязательная политика:**

- В логах фиксировать только **факт** шага (например, `onboarding_step=redmine_key_received`, `room_id`/`event_id` при необходимости для отладки), **без** тела сообщения, без ключей и без полного текста пользователя на чувствительных шагах.
- Не эхоить ключ и ID в ответах бота.
- Не класть секреты в `runtime_status`, трассировки и т.п.

Redmine ID сам по себе менее критичен, чем ключ, но его тоже можно не логировать в открытом виде (или маскировать), единообразно.

### Можно ли сразу удалить сообщение из чата?

**Частично, с оговорками Matrix:**

- Операция в протоколе — обычно **redaction** события (`m.room.redaction`), а не «стирание из истории навсегда» на всех клиентах: у части клиентов остаётся placeholder, на сервере могут сохраняться метаданные в соответствии с политикой homeserver.
- Чтобы **заредактировать чужое** сообщение (сообщение пользователя с ключом), у аккаунта бота в комнате нужен **достаточный power level** (часто в DM оба участника с одним уровнем — тогда бот **не сможет** удалять чужие события без смены правил комнаты при создании / без модератора). Это нужно **проверить** на вашем Synapse/конфиге и при необходимости задокументировать: «после отправки ключа пользователь может удалить сообщение сам» или поднять PL бота в шаблоне DM.
- Итог: **пытаться** сделать redaction сразу после обработки — разумно, но рассчитывать на это как на гарантию нельзя; основная защита — **не хранить секрет в логах** и **шифровать в БД**.

### «Захешировать в БД» — подходит ли хеш для API-ключа?

**Нет, если бот потом должен ходить в Redmine от имени пользователя.**

- **Хеш** (например SHA-256) **односторонний**: по значению в БД ключ восстановить нельзя → запросы к Redmine с этим ключом невозможны.
- **Хеш** уместен для: подтверждения «мы уже видели такой токен», audit-only, или отдельного сценария без хранения ключа (тогда Redmine вызывается только глобальным ключом сервиса — это уже другая архитектура).
- Для хранения персонального API-ключа нужно **шифрование с возможностью расшифровки** (например через существующий `APP_MASTER_KEY` / те же примитивы, что для секретов в админке), поле в БД — ciphertext, не логировать, не отображать в UI.

Кратко: **не логировать** — да; **удалить/заредактировать сообщение** — желательно, но не везде технически и не на 100%; **только хеш ключа в БД** — нет для сценария «бот использует ключ пользователя» — нужно **шифрование**.

## Интерактивность в Matrix

Универсальных чекбоксов в DM нет:

- нумерованные меню и ответы вида `1,3,5` или по одному шагу;
- state machine на паре **`(sender_mxid, room_id)`** (или только `room_id` в чистой DM).

## Надёжность state machine

Состояние **лучше хранить в Postgres**, а не только в памяти процесса (рестарт бота не должен «ломать» диалог).

Пример таблицы **`onboarding_sessions`** (черновик до финального commit в `bot_users`):

| Поле | Назначение |
|------|------------|
| `sender_mxid`, `room_id` | Ключ сессии (уникальная пара) |
| `current_step` | Имя шага |
| `draft_json` / отдельные колонки | Несекретные черновики; **секреты в черновике не хранить дольше необходимого** — после ввода ключа сразу валидировать, шифровать, сохранять в целевое поле, обнулять из draft |
| `updated_at` | Таймаут простоя (например 15–30 мин) → сброс или предложение `!start` |
| `created_at` | Аналитика / очистка |

Команды **`!cancel`** — сброс сессии; **`!change`** — повторный проход полей с существующей записью `bot_users`.

## Валидация Redmine

Перед записью в прод:

- запрос к Redmine с переданным ключом (`users/current.json` или эквивалент);
- сравнение **`user.id`** с введённым числовым Redmine ID;
- таймаут HTTP, понятные сообщения об ошибках пользователю;
- ошибки валидации логировать **без** ключа (код ответа, тип ошибки).

## Конфликт: запись от админа vs самообслуживание

Если для `redmine_id` уже есть `bot_users`:

- политика на выбор: **запрет перезаписи** с уведомлением админа / пользователю «обратитесь к администратору»;
- или **явная перезапись** только с подтверждением или флагом «источник: matrix_onboarding»;
- зафиксировать решение в коде и в `ADMINISTRATOR_GUIDE.md`.

## Структура модуля (ориентир)

```
src/matrix_onboarding.py
├── шаги (enum / константы)
├── загрузка/сохранение сессии в БД
├── обработка команд !start / !cancel / !change / !help / !status
├── обработка текстовых ответов по текущему шагу
├── validate_redmine_key_and_user_id(...)
├── шифрование ключа (общие утилиты с админкой по возможности)
└── финальный upsert bot_users
```

## Безопасность (сводка)

- Ключ в истории Matrix остаётся риском на стороне клиента и сервера; redaction и отсутствие логов **снижают**, но не убирают риск полностью.
- Предупреждение пользователю в тексте бота; для высоких требований — одноразовые токены / выдача ключа вне чата.
- Ограничение длины и формата ввода; никогда не возвращать ключ в ответах.

## Связь с админкой

- Админ по-прежнему может создавать и править **`bot_users`** вручную.
- Глобальные и групповые маршруты (статусы, версии) не зависят от способа онбординга.
