# План улучшений: version-routing и admin UI (финальная версия)

## Что уже учтено из правок

- Принцип `bugfix -> tests -> refactor` сохранен как базовый порядок.
- Явный запрет fallback для `Передано в работу.РВ` без версии закреплен.
- Единая нормализация `normalize_version_key()` вынесена в обязательную практику.
- Runtime unit-тесты и CI smoke миграций на чистой БД включены в план.

## Цель

Минимизировать риск инцидентов при доработке маршрутизации версий: обеспечить предсказуемое поведение, безопасную миграцию данных и быструю диагностику.

## KPI успеха

- 0 ложных отправок для `Передано в работу.РВ` без версии.
- 100% coverage новых endpoint-ов version-routes: happy-path + error-path.
- Время диагностики "почему выбрана эта комната" <= 10 минут.
- CI стабильно проходит `upgrade -> downgrade -> upgrade` на fresh БД.
- После релиза доля `no_route` контролируется метрикой и не растет скачкообразно.

## Критические риски и контрмеры

### Риск 1: Потеря данных при миграции нормализации
- **Проблема:** при коллизиях (`"1.0"` и `"1.0 "`) обычный `UPDATE` может вызвать конфликт unique или потерю бизнес-смысла.
- **Контрмера:**
  - делать миграцию идемпотентной;
  - до обновления собирать коллизии в таблицу `version_route_conflicts`;
  - нормализовать только безопасные строки;
  - конфликтные записи отправлять на ручной разбор.

### Риск 2: Расхождение нормализации между Python и SQL
- **Проблема:** разные правила в runtime/admin/миграции приводят к "плавающему" матчингу.
- **Контрмера:**
  - единая функция `normalize_version_key()` как source of truth для Python;
  - SQL-нормализация максимально эквивалентна Python-логике;
  - тесты на edge-cases: `None`, пробелы, `\t\n\r`, mixed case, emoji.

### Риск 3: Неявный fallback в роутинге
- **Проблема:** fallback может отправлять события в неверную комнату.
- **Контрмера:** явно закрепить стратегию:
  - приоритет `user > group > global`;
  - fallback только для `new`;
  - `РВ` без версии -> `no_route` + warning.

### Риск 4: Недостаточная наблюдаемость
- **Проблема:** текстовые логи сложны для поиска аномалий и алертинга.
- **Контрмера:**
  - структурированный JSON-лог `route_resolved`;
  - обязательные поля: `source`, `rule_id`, `issue_id`, `user_id`, `version_normalized`, `room_id`;
  - счетчик метрик по `source` и `matched`.

## Этап 1: критичное (День 1)

### 1) Исправить fallback для `РВ` без версии
- **Приоритет:** высокий
- **Оценка:** 2-3 часа
- **Действия:**
  - Явное правило: `status == RV && empty(version) -> no_route`.
  - Лог warning с контекстом (`user_id`, `status`, `version`).
  - Unit-тест регрессии.

### 2) Внедрить `normalize_version_key()` в runtime/admin
- **Приоритет:** высокий
- **Оценка:** 4-6 часов
- **Действия:**
  - helper в `src/utils/version_key.py`;
  - применять при сохранении route и при матчинге версии;
  - добавить docstring с примерами.

### 3) Параметризованные unit-тесты логики маршрутизации
- **Приоритет:** высокий
- **Оценка:** 3-5 часов
- **Действия:**
  - кейсы `user/group/global/fallback/no_route`;
  - отдельно проверить `РВ` без версии;
  - проверять `source`, не только room.

## Этап 2: стабилизация (День 2)

### 4) API-тесты endpoint-ов version-routes
- **Приоритет:** высокий
- **Оценка:** 6-8 часов
- **Действия:**
  - покрыть add/delete для users и groups;
  - сценарии: `admin`, `non-admin`, `invalid-key`, `duplicate`, `no-room`, `csrf`.

### 5) CI smoke миграций на чистой БД
- **Приоритет:** средний
- **Оценка:** 3-5 часов
- **Действия:**
  - отдельный job с fresh PostgreSQL;
  - `upgrade head -> downgrade base -> upgrade head`;
  - артефакты/логи при падении.

### 6) Структурированные логи и базовая метрика
- **Приоритет:** средний
- **Оценка:** 3-5 часов
- **Действия:**
  - JSON-лог `route_resolved`;
  - счетчик `routing_events_total{source,matched}`;
  - минимальный post-release алерт на рост `no_route`.

## Этап 3: безопасная миграция и документирование (День 3)

### 7) Миграция нормализации с карантином конфликтов
- **Приоритет:** высокий
- **Оценка:** 1 день
- **Действия:**
  - создать `version_route_conflicts` (idempotent);
  - записывать туда конфликтующие ключи до `UPDATE`;
  - нормализовать только неконфликтные записи;
  - после очистки добавить unique-ограничение по нормализованному ключу;
  - после деплоя обязательная ручная проверка конфликтов.

### 8) Документировать стратегию в `docs/ROUTING_STRATEGY.md`
- **Приоритет:** высокий
- **Оценка:** 0.5-1 день
- **Действия:**
  - приоритеты маршрутов, fallback-правила, примеры конфликтов;
  - связь с `README.md` и `docs/ADMINISTRATOR_GUIDE.md`.

## Этап 4: UX и эксплуатационные улучшения (День 4+)

### 9) Dry-run симулятор `/routes/simulate`
- **Приоритет:** высокий
- **Оценка:** 2-4 дня
- **Действия:**
  - endpoint возвращает `room_id/source/rule_id/matched`;
  - гарантировать отсутствие отправки уведомлений (`mock/send-disabled` в тестах);
  - простая UI-форма для ручной проверки.

### 10) Audit trail изменений маршрутов
- **Приоритет:** средний
- **Оценка:** 2-3 дня
- **Действия:**
  - записывать `кто/когда/старое->новое`;
  - использовать существующую таблицу аудита с `entity_type=version_route` (или эквивалент).

### 11) Импорт/экспорт конфигурации
- **Приоритет:** средний
- **Оценка:** 3-5 дней
- **Действия:**
  - сначала read-only экспорт;
  - импорт через dry-run + подтверждение.

## Риски и зависимости

- Коллизии после нормализации потребуют ручного решения владельцем конфигурации.
- Изменение приоритетов роутинга может изменить текущий поток уведомлений.
- Для метрик/алертинга может потребоваться инфраструктурная поддержка (Prometheus/Loki/ELK).
